# Legacy rule audit and migration

## Type

AFK

## User stories covered

15

## What to build

Audit all 27 original compression rules and explicitly migrate each one into the appropriate new category or retire it. Remove the temporary legacy phase from the engine once migration is complete. Update existing test fixture expected outputs to reflect any changes from the migration.

Each legacy rule should be handled with an explicit decision: if the pattern maps to a new category (e.g., a wordy-phrase rule fits Phase 4), move it there. If the pattern is too specific to justify keeping (e.g., `"The goal of this tool is to"`), retire it with a brief note explaining why.

After this slice, the engine should have no legacy phase — all rules live in their proper category, and the rule set is clean and auditable.

## Acceptance criteria

- [ ] Every one of the original 27 rules has an explicit disposition: migrated to a named category or retired with a reason.
- [ ] The temporary legacy phase is removed from the engine.
- [ ] Migrated rules function correctly in their new category and are covered by that category's tests.
- [ ] Existing golden-fixture expected outputs are updated to reflect the final rule set (re-baselined, not constraining new rules to match legacy expectations).
- [ ] All tests pass after migration, including any threshold assertions on the re-baselined fixtures.

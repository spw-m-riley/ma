# Phased rule engine with contraction rules and negation safety

## Type

AFK

## User stories covered

1, 4, 6, 7, 8, 10, 12, 15, 16

## What to build

Replace the single-pass rule application in the compression engine with a multi-phase architecture that applies rule categories in a strict, defined order. Add contraction rules (~15) as Phase 1 — the first real category in the new engine. Existing rules should remain functional in a temporary legacy phase so that current behavior is preserved until later slices migrate or retire them.

The phase ordering is the critical architectural decision: contractions must run before any future subject-elision rules to prevent negation inversion. The engine should enforce ordering structurally (each phase is a distinct unit that exports its rules, and the engine calls them in sequence) rather than relying on rule definition order within a flat list.

Each rule category should live in its own compilation unit so that categories can be added, tested, and maintained independently. The registry composes all category functions into an ordered pipeline.

## Acceptance criteria

- [ ] The compression engine applies rules in distinct, ordered phases rather than a single flat pass.
- [ ] Contraction rules (~15) are implemented as Phase 1 and correctly transform negation patterns: `do not` → `don't`, `should not` → `shouldn't`, `cannot` → `can't`, etc.
- [ ] A dedicated negation-inversion test proves that `"You should not delete files"` produces `"You shouldn't delete files"` and never `"delete files"`.
- [ ] Existing rules remain functional in a temporary legacy phase; current test fixtures continue to pass (with expected outputs updated only where contraction rules legitimately change them).
- [ ] Running `ma compress` on a real instruction file that contains `do not`, `should not`, or similar patterns produces non-zero token reduction.
- [ ] Zone protection is unaffected: headings, code fences, inline code, URLs, and file paths pass through unchanged.
- [ ] Rule categories are defined in separate compilation units, each exporting a function that returns its rule set.
- [ ] All existing tests pass after the restructure.

# Demonstrative pruning and technical abbreviations

## Type

AFK

## User stories covered

2, 5

## What to build

Add Phase 7 (demonstrative/determiner pruning, ~11 rules) and Phase 8 (technical abbreviations, ~16 rules) to the phased compression engine. This slice is responsible for making even already-terse technical instruction files show non-zero reduction.

Demonstrative pruning removes redundant determiners and demonstratives: `"This is the"` → `"the"`, `"The following"` → `"these"`, `"the existing"` → `"existing"`, `"the current"` → `"current"`, `"the same"` → `"same"`.

Technical abbreviations shorten commonly abbreviated technical terms to their standard short forms: `"configuration"` → `"config"`, `"repository"` → `"repo"`, `"environment"` → `"env"`, `"authentication"` → `"auth"`, `"documentation"` → `"docs"`, `"implementation"` → `"impl"`, `"application"` → `"app"`, etc. These rules should use word-boundary anchors to prevent partial-word matches.

Research measured abbreviation rules alone delivering 3–8% reduction on technical prose. Per-category tests should include word-boundary edge cases and confirm that abbreviations do not fire inside code spans or headings (already handled by zone protection, but worth asserting).

## Acceptance criteria

- [ ] Demonstrative pruning rules (~11) are implemented as Phase 7 and correctly reduce redundant determiners.
- [ ] Technical abbreviation rules (~16) are implemented as Phase 8 and correctly shorten long-form technical terms to standard abbreviations.
- [ ] Abbreviation rules use word-boundary anchors and do not produce partial-word matches.
- [ ] Per-category unit tests exercise representative patterns for both categories, including word-boundary edge cases.
- [ ] Running `ma compress` on a terse instruction file that contains terms like `configuration`, `repository`, and `environment` produces non-zero token reduction.
- [ ] Zone protection ensures abbreviations do not fire inside code fences, inline code, headings, URLs, or paths.
